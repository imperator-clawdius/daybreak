# Daybreak sale-readiness premortem

Goal: make Daybreak feel like a professional, sale-ready shipped product while
keeping the readiness gate honest. No fabricated proof, payment links, order
counts, screenshots, or testimonials.

## Failure modes closed in this hardening pass

| Risk | Why it matters | Mitigation |
| --- | --- | --- |
| Carried items duplicate across days | A repeated unresolved item would make the lock screen feel broken and punitive. | `carryForward()` now keeps only the latest copy per item id and does not resurrect later-resolved items. |
| Evening review rewrites morning completion | Streak and completion state could drift if evening persistence recalculated the wrong flag. | `resolveLogForPhase()` updates only the active ritual flag, and the renderer uses it. |
| Junk commitments enter the board | Empty, duplicate, or over-limit commitments weaken the core "three real commitments" promise. | Core `validateNewCommit()` normalizes text, rejects duplicates, enforces length, and blocks a fourth live commitment. |
| Commit gesture cannot finish the morning ritual | If "commit" leaves an item indistinguishable from an untouched item, the user can only close Daybreak by deferring or killing everything. | Core now distinguishes morning `pending` items from kept `open` commitments; `commit` moves an item to `open`, morning dismissal waits only on `pending`, and same-day reloads reuse the saved board without incrementing carry count. |
| Renderer payload bypasses the close invariant | A renderer bug or malformed IPC payload could drop pending items and ask main to close with a board the user never wiped. | Core `validateLogUpdate()` rejects dropped existing items, mutated immutable item fields, invalid phase states, evening additions, and over-cap new commitments; main now accepts save/dismiss only against its active session. |
| Desktop renderer escapes the local shell | Unexpected navigation or a new-window attempt would weaken the local-only, no-browser-surface promise of the Windows app. | The BrowserWindow now runs with `sandbox: true`, denies `window.open`, and uses a core-tested navigation policy that allows only the packaged `index.html` entrypoint. |
| Desktop renderer silently relaxes CSP | A local Electron shell without a strict policy can drift toward loading inline scripts, remote assets, form posts, or object/embed content even while navigation is blocked. | Core now owns the desktop renderer CSP, `desktop/src/renderer/index.html` uses that exact policy, and `scripts/desktop-renderer-security.test.ts` rejects policy drift. |
| No-telemetry promise regresses silently | The landing page promises no telemetry; a later dependency, runtime API call, or remote desktop asset could violate that without changing the core ritual. | Added `npm run verify:local-only`, which scans desktop runtime source/static assets for outbound network/telemetry APIs or remote URLs and desktop runtime dependencies for telemetry packages before the no-telemetry promise is considered guarded. |
| App does not actually open at first login | The product promise says Daybreak takes over at first login; without Windows startup registration it only runs when manually launched. | Packaged Windows production builds now call `app.setLoginItemSettings({ openAtLogin: true })` through a core-tested policy; smoke and unpackaged dev runs do not mutate login items. |
| Interrupted JSON write loses local commitments | Directly overwriting the local JSON file could leave a corrupt primary file and force Daybreak to fall back to an empty ritual. | Desktop `Store` now writes through a same-directory `.tmp` file, keeps the previous valid store as `.bak`, and reads the backup if the primary JSON is corrupt. |
| Malformed local JSON crashes the ritual | A hand-edited or partially corrupted file with `version: 1` and a `days` array could pass the old shallow check and break session construction. | Core now validates persisted day logs and items; the desktop store rejects malformed primary data and falls back to a valid backup. |
| Desktop packaging uses stale core output | Packaging the desktop workspace directly could bundle an older `@daybreak/core/dist` if the core workspace had not been rebuilt first. | `@daybreak/desktop` now builds `@daybreak/core` before esbuild bundling, and Vitest aliases `@daybreak/core` to source so tests never pass against stale dist output. |
| Built core package cannot be imported by plain Node ESM | A professional workspace package should work through its published import surface, not only through test aliases or bundlers. | Core source now uses emitted `.js` relative specifiers, and `scripts/core-package.test.ts` builds `@daybreak/core` then imports `packages/core/dist/index.js` in plain Node. |
| Product promises swipe-to-wipe but ships button-only interaction | A buyer expecting the core ritual gesture would immediately feel the app is unfinished if the desktop app only exposed small action buttons. | Core now owns a tested swipe gesture policy; the renderer supports mouse/touch pointer swipes with button fallback, and Electron smoke proves an add-and-swipe flow persists correctly. |
| Evening review works only in unit tests | The v1 loop includes an evening review; without a runtime proof, a packaging or IPC regression could ship a morning-only app. | Main-process session selection now lives in core with tests, and Electron smoke has separate morning and evening scenarios that prove swipe persistence in both phases. |
| Weekly streak claim is not backed by app history | The landing page promises a weekly streak, but a renderer that only knows today's board cannot display a real history-backed streak. | Core now exposes a tested streak summary that merges stored history with the current day; the desktop bridge passes history to the renderer, and evening smoke proves the weekly streak text renders from historical data. |
| Weekly streak counts non-consecutive weeks | Counting distinct historical weeks would overstate progress after a missed week and weaken the "honest streak" promise. | Core weekly streak math now walks backward from the current Monday-anchored week and stops at the first missed week. |
| Fake HTTPS checkout/download URLs turn gates green | A placeholder external URL could make readiness look complete without real proof. | `verify:readiness` now requires a `buy.stripe.com` Payment Link for checkout and HTTP 2xx proof for checkout/download URLs. |
| Host-only Stripe URL passes as a Payment Link | A bare `https://buy.stripe.com` host is not a checkout link, but a host-only allowlist could make the public CTA or readiness gate trust proof for a non-link URL. | Core checkout state and `verify:readiness` now require a non-root path on `buy.stripe.com` before treating a URL as a Stripe Payment Link. |
| Tokenized sale URLs leak private state or expire after launch | A copied Stripe URL with prefilled query data, a fragment, or a signed installer URL with credentials/query tokens could make the public CTA expose private state or silently expire after launch. | Core checkout/download policy and `verify:readiness` reject query strings, fragments, and embedded credentials for public sale URLs before proof can make either CTA live. |
| Wrong-price Stripe link passes checkout readiness | A live Stripe Payment Link could be active and reachable while charging the wrong amount, using test mode, or being recurring instead of one-time. | Core and readiness checks now require `proof/stripe-payment-link.json` to prove the configured Payment Link is active, live-mode, one-time, and USD 1900 cents. |
| Missing or extra Stripe line item hides checkout truth | A Payment Link proof could omit line items or contain one correct Daybreak line plus another fee, upsell, or product and still look like the advertised $19 one-time checkout. | Core and readiness checks now require exactly one line item in Stripe Payment Link proof before checkout can go live. |
| Discounted checkout masquerades as a full-price $19 sale | A Payment Link or Checkout Session proof can show a 1900-cent line item while still allowing promotion codes, coupons, discounts, or total-detail adjustments that make the buyer pay less than the advertised one-time price. | Core proof policy and `verify:readiness` now reject discount, coupon, promotion-code, and total-detail fields across checkout, installer, and first-paid-order proof artifacts before any checkout or market-signal gate can pass. |
| Malformed or partial Stripe proof crashes checkout export or verifier | A hand-built proof file with a non-object Payment Link block, numeric/missing Payment Link URL, missing/blank/non-`plink_` Payment Link ID, bare `plink_` prefix without an ID suffix, string active/live-mode flags, string/array `line_items` containers, string/number line-item pagination flags, paginated/incomplete line items, `line_items.data` empty or shaped as an object, containing non-object entries, malformed or fractional quantity fields, malformed recurring fields, malformed or fractional-cent price fields, or omitting the nested price object could throw, pass accidentally, or be misclassified during static export/readiness verification instead of failing closed. | Core and readiness checks now reject malformed Payment Link object/URL/ID/state fields and missing, incomplete, or malformed line item container, pagination, quantity, recurring, and price data with explicit pending reasons. |
| Malformed Stripe IDs pass proof checks | A hand-built proof could pad `plink_` or `cs_live_` IDs with spaces, or add punctuation/embedded spaces after the prefix, and still pass if validators only check the prefix. | Core and readiness checks now require Payment Link and Checkout Session IDs to match exact `plink_...` / `cs_live_...` shapes with only word-character suffixes. |
| Public checkout button goes live before Stripe proof exists | The readiness gate could stay honest while the landing page exposes an unproven Payment Link to buyers. | The public checkout CTA now uses core's proof-backed Stripe state, reads `proof/stripe-payment-link.json` during static export, and stays in the "opening soon" state until the configured link is active, live-mode, one-time, and USD 1900 cents. |
| Legal pages keep saying checkout is closed after Stripe goes live | Terms and privacy copy can become stale when checkout readiness changes, undermining trust at the exact moment buyers need policy clarity. | Terms and privacy pages now read the same proof-backed public checkout state as the CTA; `scripts/legal-checkout-copy.test.ts` rejects hard-coded checkout-unavailable copy. |
| Corrected Stripe proof does not republish the checkout CTA | The public checkout state is computed at static export time, so a proof-only correction could leave the deployed page stale if Pages does not rebuild. | The Pages workflow now triggers on `proof/stripe-payment-link.json`, with a regression test proving that proof changes redeploy the static site. |

| Valid signature from the wrong publisher passes release | Authenticode can be valid while still not proving the installer was signed by the Daybreak publisher. | Release preflight now rejects valid signatures unless the signer subject includes `Passive Print Labs LLC`. |
| Valid signature without timestamp passes release | Authenticode can be valid today without timestamp proof but become unreliable after certificate expiry. | Release preflight, readiness, and public download proof now require timestamped signatures before treating an installer as releasable. |
| Preview health gets confused with production HTTPS | During certificate issuance, the project Pages preview may redirect to the custom domain, and a redirect-following verifier can accidentally borrow the apex response. | Launch verification now reports `PREVIEW_SITE` separately from `LIVE_SITE` and `APEX_SITE`, and preview checks use manual redirects so a custom-domain redirect reports pending instead of pass. |
| HTTP apex success masks HTTPS failure | GitHub Pages can serve the apex over HTTP while the certificate is still wrong, which looks "live" in a browser but is not production-ready. | Launch verification now reports `APEX_HTTP_SITE` separately from the HTTPS `APEX_SITE` gate. |
| WWW domain breaks silently | Buyers may type `www.daybreak.rest`; checking only the apex would miss a broken or certificate-pending `www` redirect. | Launch verification now reports `WWW_DNS`, `WWW_HTTP_SITE`, and `WWW_SITE`, and default production launch stays pending unless `www` HTTPS also serves Daybreak. |
| WWW legal/crawler routes break silently | A `www` homepage could work while `/privacy/`, `/terms/`, `robots.txt`, or `sitemap.xml` fail for buyers and crawlers. | Launch verification now checks `WWW_HTTP_ROUTES` and `WWW_ROUTES` with the same required-route policy used for the apex. |
| Readiness domain gate ignores WWW | If readiness checked only the apex, it could advance the domain gate while `www.daybreak.rest` still failed for buyers. | The production-domain readiness gate now evaluates both `daybreak.rest` and `www.daybreak.rest`, accepts GitHub Pages IPv6 DNS for the CNAME, and keeps the gate pending until both hosts serve Daybreak over HTTPS. |
| Pages certificate state is manually inspected | Without a repeatable Pages-health command, certificate issuance can regress into screenshots or vague dashboard checks. | Added `npm run verify:pages-health`, which polls GitHub's Pages DNS health endpoint, uses local `gh` auth or env tokens, reports apex and `www` DNS/Pages/HTTPS flags, and exits pending until the certificate is approved and HTTPS is enforced. |
| GitHub Pages health lags buyer-visible redirects | GitHub's Pages health object can report `www.enforces_https=false` even after the actual edge redirects HTTP and `www` traffic to the HTTPS apex, leaving a false pending gate. | Pages-health now fetches manual edge redirects for apex HTTP, `www` HTTP, and `www` HTTPS canonicalization, prints that evidence, and accepts it only when DNS, Pages serving, certificate approval, HTTPS response, and actual redirects all prove the buyer-visible path is ready. |

Run these after each sale-readiness change:

```powershell
npm run check

npm run verify:readiness

npm run verify:pages-health
$env:DAYBREAK_SMOKE = "1"; npx electron .; Remove-Item Env:DAYBREAK_SMOKE
```

exist and are verified. `verify:release` should keep exiting 1 until the
